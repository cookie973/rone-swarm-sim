  SITL 实例号: 5
  Bridge RX: 14506

说明：

3. 这样做的目的，是在不破坏 QGC 常规控制的情况下，让 ROS2 节点同时观察并处理蜂群级命令。

八、起飞策略优化说明

文件：
/home/cookie/gz_ws/src/ardupilot_gazebo/scripts/takeoff_all.sh
/home/cookie/gz_ws/src/ardupilot_gazebo/scripts/takeoff_all.py

当前版本的起飞策略已经切换到“shell 封装 + ros2 CLI 逐机顺序起飞”作为默认路径，Python 协调器保留为可选回退路径。

核心策略：
1. takeoff_all.sh 负责准备 ROS 环境，并默认执行已验证通过的 CLI 起飞流程。
2. 为避免与 `swarm_formation` 的速度控制冲突，`takeoff_all.sh` 默认会在起飞前暂停 `/swarm_formation`。
3. 对每架无人机按状态闭环推进：先低油门 RC override 保持可解锁，再发 GUIDED 和 ARM，等待 armed=true。
4. 解锁成功后切换到高油门 RC override，并持续补发 TAKEOFF 指令直到高度收敛。
5. 成功判据是实际高度达到目标附近，而不是只看命令 ACK。
6. 每架机起飞完成后立即释放 RC override，再推进下一架，避免多机互相污染油门状态。
7. 如需回退旧的 Python 协调器，可设置 TAKEOFF_ENGINE=python。

默认优化目标：
1. 尽量让 6 架机在更短时间内进入同步起飞阶段。
2. 尽量降低因个别飞机慢响应导致整队变慢的情况。
3. 尽量降低“有的飞机掉锁后整队又停住”的情况。

可调环境变量：
1. NUM_UAV
2. SUCCESS_ALTITUDE
3. PER_UAV_TIMEOUT
4. TAKEOFF_STREAM_DURATION
5. RC_OVERRIDE_RATE
6. RC_OVERRIDE_THROTTLE
7. RC_OVERRIDE_IDLE_THROTTLE
8. TAKEOFF_STOP_CONTROLLER（默认 1；设为 0 可跳过自动暂停编队控制器）
9. TAKEOFF_ENGINE（默认 cli；设为 python 可回退旧协调器）

示例：
PER_UAV_TIMEOUT=30 RC_OVERRIDE_THROTTLE=1620 ./takeoff_all.sh 6.0

推荐用法：
1. 正常实验直接执行：
   ./takeoff_all.sh
2. 指定高度执行：
   ./takeoff_all.sh 8.0
3. 如果你发现个别机器解锁偏慢，可以适当增大 PER_UAV_TIMEOUT。
4. 如果你需要兼容旧逻辑，可以显式使用 TAKEOFF_ENGINE=python ./takeoff_all.sh。

九、QGroundControl 集成方式

QGC 接入端口：
1. uav1 -> UDP 14540
2. uav2 -> UDP 14541
3. uav3 -> UDP 14542
4. uav4 -> UDP 14543
5. uav5 -> UDP 14544
6. uav6 -> UDP 14545

推荐用法：
1. 在 QGC 中分别添加 6 个 UDP 连接。
2. 目标地址使用 127.0.0.1。
3. 端口使用 14540 到 14545。
4. 每个连接会对应一架无人机，SYSID 由 SITL 的 --auto-sysid 自动隔离。

QGC 详细配置步骤：
1. 先启动仿真栈。
   推荐今天使用：
   `cd /home/cookie/gz_ws/src/ardupilot_gazebo/scripts`
   `./swarm_start.sh --no-controller`
2. 等待 Gazebo、6 个 SITL、6 个 bridge、1 个 `qgc_bridge` 全部起来。
3. 打开 QGroundControl。
4. 进入 QGC 顶部菜单 `Application Settings`。
5. 打开 `Comm Links`。
6. 点击 `Add` 新建链路。
7. 类型选择 `UDP`。
8. 名称建议依次填写：`uav1`、`uav2`、`uav3`、`uav4`、`uav5`、`uav6`。
9. 每个链路都按下面方式填写：
   `Host Address`: `127.0.0.1`
   `Port`: 对应本机 UDP 端口
   `Auto Connect`: 打开
10. 六个连接建议按下表配置：

   `uav1 -> 127.0.0.1:14540`
   `uav2 -> 127.0.0.1:14541`
   `uav3 -> 127.0.0.1:14542`
   `uav4 -> 127.0.0.1:14543`
   `uav5 -> 127.0.0.1:14544`
   `uav6 -> 127.0.0.1:14545`

11. 保存后返回主界面，等待心跳出现。
12. 如果只看到 1 架机，先不要改仿真，先检查 `Comm Links` 里是否真的保存了 6 条 UDP 链路。
13. 如果 6 条都在但仍未全部显示，执行一次：
   `./swarm_logs.sh qgc`
   观察是否出现 `QGC 客户端已连接` 的日志。
14. QGC 中每架机应有独立 SYSID：
   `uav1=1`、`uav2=2`、`uav3=3`、`uav4=4`、`uav5=5`、`uav6=6`
15. QGC 里不要直接连接 `14740-14745`，这些是内部中继端口，只给 `qgc_bridge` 使用。
16. QGC 里也不要再额外连接 `14501-14506` 或 `14601-14606`，否则容易和现有链路打架。

QGC 连接成功后的建议检查：
1. 先确认 QGC Vehicle Setup 里能切到 6 架不同飞机。
2. 先单独看每架机的姿态、GPS、模式和电池页面是否刷新。
3. 今天只建议做单机模式切换、单机任务查看、单机状态监控。
4. 如果要做全队起飞，优先使用脚本 `./takeoff_all.sh 5.0`，不要先依赖 QGC 广播命令。

当前 QGC 集成分成两类能力。

第一类：单机直控
1. QGC 对任意单架无人机的常规操作，会通过 qgc_bridge 原样转发到对应 SITL。
2. 包括常规的模式切换、起飞、降落、航点飞行和遥控指令。
3. 只有在你显式启动 `swarm_formation` 且其工作正常时，leader_follower 才会带动其余飞机跟随。

第二类：蜂群级命令
1. `qgc_bridge` 支持把一部分 MAVLink 命令翻译成 ROS2 蜂群命令。
2. 这类命令适合从 QGC 的高级命令面板、自定义命令入口或原始 MAVLink 命令工具发送。
3. 但今天不建议把它作为编队演示入口，因为编队切换策略仍在重写中。

自定义蜂群命令约定：
1. MAV_CMD_USER_1
   用途：切换编队
   param1:
   0 -> circle
   1 -> line
   2 -> v_shape
   3 -> grid

2. MAV_CMD_USER_2
   用途：切换蜂群模式
   param1:
   0 -> anchored
   1 -> leader_follower

3. MAV_CMD_USER_3
   用途：批量 arm/disarm
   param1:
   0 -> disarm all
   1 -> arm all

4. MAV_CMD_USER_4
   用途：批量起飞/降落
   param1:
   0 -> land all
   1 -> takeoff all
   param2:
   起飞高度，单位米

广播命令支持：
1. 如果 COMMAND_LONG 的 target_system 为 0 或 255，qgc_bridge 会把以下命令视为全体广播：
   MAV_CMD_COMPONENT_ARM_DISARM
   MAV_CMD_NAV_TAKEOFF
   MAV_CMD_NAV_LAND
2. 如果 SET_MODE 的 target_system 为 0 或 255，qgc_bridge 会把该模式同步发布到所有无人机。

十一、故障排查

问题 1：swarm_start.sh 启动失败
排查：
1. 先看 gazebo.log。
2. 再看 uav1.log。
3. 检查 /opt/ros/humble/setup.bash 是否存在。
4. 检查 $HOME/ardupilot/Tools/autotest/sim_vehicle.py 是否存在。

问题 2：QGC 连不上无人机
排查：
1. 查看 qgc_bridge.log。
2. 确认 swarm_start.sh 已经启动 qgc_bridge。
3. 确认 QGC 连接的端口是 14540-14545，而不是 14740-14745。
4. 确认本机没有旧的 QGC 或其他 GCS 占用这些端口。
5. 确认 `swarm_start.sh` 使用的是当前版本，里面的 SITL 启动参数包含 `--mavproxy-args="--non-interactive --streamrate=20"`。

问题 3：批量起飞仍然不整齐
排查：
1. 查看各自 uavN.log 中是否已经拿到 heartbeat。
2. 查看 bridge_uavN.log 中是否已经收到 mode 和 armed 状态。
3. 查看脚本汇总里失败的是 GUIDED、ARM 还是 TAKEOFF 阶段。
4. 确认 EKF/GPS 已经就绪。
5. 对偏慢机器可单独查看：
   ./swarm_logs.sh sitl N
   ./swarm_logs.sh bridge N

问题 4：编队切换还是乱、看不出形状
说明：
1. 这是当前尚未解决的问题。
2. 今天先不要继续把它当成可用功能。
3. 明天从“重新设计切换执行策略”开始处理，而不是继续小幅调参。

问题 5：QGC 只能看到部分飞机
排查：
1. 先检查 QGC `Comm Links` 里是否确实有 6 条 UDP 链路。
2. 再看 `./swarm_logs.sh qgc` 是否打印了每个 `uavN QGC 客户端已连接`。
3. 如果只有部分端口被连接，删除 QGC 旧链路后重新按 14540-14545 添加。
4. 确认没有把内部端口 14740-14745 配进 QGC。

问题 6：旧进程干扰新启动
处理：
1. 先执行 ./swarm_stop.sh。
2. 如果仍有残留，再手工执行：
   pkill -9 -f 'gz sim'
   pkill -9 -f arducopter
   pkill -9 -f mavproxy
   pkill -9 -f mavlink_tx
   pkill -9 -f qgc_bridge
